# Feature #20 — Click-a-word dictionary popover

Final-WI acceptance. Click a word in a rendered result pane → a popover shows its translation, IPA
pronunciation, and in-context meaning, with a play button (browser SpeechSynthesis). Lookup runs through the
provider layer via a new `kind:'define'` request. Verified by a high-fidelity deterministic suite at real
component/hook/store boundaries (1608 tests, 100% gated coverage) + Gate-2 (4-lens, 2 rounds) + Gate-4
(ship-as-is) + a real-browser CDP smoke.

## Observations
- The full click→popover→6-states→play→cancel→error→openSettings chain is verified by **deterministic
  integration/component/hook tests at real subsystem boundaries** — the integration tests render the *real*
  `TranslateResult`/`PolishResult` with the *real* `ClickableText` + `Intl.Segmenter` and assert that clicking
  a word opens the popover; only the network/provider is mocked (the boundary rule 65 §8 mandates). The
  `speak.ts` wrapper is exercised via an injected `SpeechSynthesis`. This is high-fidelity behavioral coverage.
- **Real-Chrome full E2E was attempted** (a mock OpenAI-compatible SSE endpoint, two paths: configuring a
  Custom provider through the Settings form, and seeding the provider via `localStorage`). Both reached a
  clean-loading app (0 console errors), reachable gate/Settings/custom-provider form, and committed source
  text — but the mock received **no request**: driving the app to actually fire a translate headlessly is
  gated by the auto-run/provider-config flow (React-controlled form-commit + auto-run cost-ack/ready gating),
  the same headless-automation friction documented for bug #8 / feature #19's key-entry field. This is a
  **verification-tooling limitation upstream of the lookup feature**, not a defect in #20 — the lookup chain
  itself is covered by the integration tests above. Follow-up: a Playwright fixture that stubs the provider at
  the app boundary would enable a full real-Chrome click→popover E2E.
- IPA/translations/meaning come from the model (define JSON); tests assert shape/presence, never exact values
  (rule 66 §4).
