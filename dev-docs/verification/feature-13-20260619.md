# Feature #13 verification — auto-expanding editor panes

Gate-5b final acceptance for the final (only) WI. The implementation is CSS-only (className changes), so
the behavioral verification is a browser pass — auto-grow is layout-engine behavior the jsdom test runner
cannot measure (rule 10). Run against the merge-commit code (`cd937c2`, v0.10.0) via `vite dev` on
`http://localhost:5173`, driven by Google Chrome `--headless=new` over raw CDP (Node built-in WebSocket).
The verification ran on the WI branch whose code is byte-identical to the clean merge commit.

## Acceptance criteria → evidence

| Design behavior (committed bundle) | Result | Evidence (measured via CDP `getBoundingClientRect`) |
|---|---|---|
| Editors grow to fit content (no inner scroll below the cap) | pass | Translate Source textarea **91px → 490px** on a 14-line paste; Polish Original **88px → 422px** (card 142→475px). The cards grow with the content. |
| Resting min-height ≈ 130px | pass | Empty Source card **129px**; empty Original/Draft cards **142px** (= header ≈ 54px + 88px textarea-min; ≥ the 130px floor, per plan LOW-1). |
| Max-height cap ≈ 88vh → inner scroll returns | pass | A 120-line paste caps the Source textarea at **803px** (viewport innerHeight 913 → 88vh ≈ 803); the textarea inner-scrolls instead of growing unbounded. |
| Translate panel grows without overlapping the Polish panel | pass | After the `shrink-0` fix: translate `<section>` **bottom 695px**, Polish `<section>` **top 695px** (no overlap). Before the fix the section stayed 342px while the 490px Source overflowed to bottom 694px (onto Polish) — caught + fixed during this verification. |
| Polish column reflow — cards keep content height, the column scrolls, Keywords reachable | pass | Original grew to 475px inside PolishPanel's existing `overflow-auto` column; the `shrink-0` cards do not compress; the `flex-none` Keywords card stays at the column's scrollable bottom (design Section D). |
| Script-aware (CJK / RTL / mixed) | pass (by mechanism) | The editors retain `dir="auto"` + `unicodeBidi: plaintext` (polish) / `srcBidi` (translate), unchanged by this diff; `field-sizing-content` measures the wrapped, per-script height — the same mechanism verified growing above. |

## Method

- `pnpm dev` (Vite, port 5173). `/tmp/verify-13.mjs`: launches `--headless=new` with `--remote-debugging-port`,
  attaches via CDP, navigates to the app, sets editor values through the native
  `HTMLTextAreaElement.prototype.value` setter + an `input` event (so React's controlled `onChange` fires),
  and measures `getBoundingClientRect()` heights + `<section>`/`<main>` geometry. Screenshots captured at
  `/tmp/13-{empty,source-grown,source-capped,polish-grown}.png` (ephemeral).
- The `pnpm check:all` gate (lint + typecheck + 100%-gated coverage + build) is green; the existing
  `PolishPanel.test.tsx` / `TranslatePanel.test.tsx` pass unchanged (they assert ARIA/value/`dir`, nothing
  height/flex-dependent).
- Gate-4 independent audit: ship-as-is, 0 Critical/High/Medium.

## Deferred (out of scope, accepted)

- **Top-fade affordance** (design depicts a scrolled-above fade) — a correct fade needs scroll-position JS,
  which would make this CSS-only WI a JS+tests change; the native scrollbar already signals the cap.
  Deferred to a follow-up (plan "Known limitations").
- **Auto-scroll-into-view during streaming-fill** — when "Translate original" streams into the Draft, the
  card grows per chunk and the column reflows; keeping the streamed caret in view is a separate concern,
  not in scope for this sizing change (the user can scroll the column).
