# Feature #26 — Tighter editor resting height

Final-WI acceptance for the 1-WI feature (`min-h-[88px]`→`min-h-[56px]` textarea, `min-h-[130px]`→
`min-h-[98px]` polish card, via the shared `src/lib/editor/editorSizing.ts` constants). Design bundle
`dev-docs/designs/lucid-editor-resting-height/` (issue #219). Feature code at commit `8b15b97`
(branch `feat/26-editor-resting-height`; branch HEAD at verify time was `7ba0634`, a docs-only commit
atop the feature code). Not yet merged; #26 version bump pending (PR tail commit).

**Result: partial.** The core resting-height change (56px/98px) is correctly implemented and verified
across all three editors + two polish cards, and grow-to-content, the 88vh cap, dark, RTL, and no-overflow
all pass. **One explicit design/plan acceptance criterion fails for the Translate Source editor only:**
"Empty and one-line rest at the same height (no jump on the first keystroke)" — see the Discovered defect
section. Recommend NOT flipping to `VERIFIED` until the Source first-keystroke jump is resolved.

## Discovered defect — first-keystroke jump on the Translate Source editor

The design and plan state, three times, that "empty and one-line rest at the **same** height (no jump on
the first keystroke)" and that the editors "rest at 56px inner for empty/one-line content". The **Source**
editor violates this:

- Empty Source renders at **91px**, not 56px. After typing one character it collapses to **57px** — a
  visible **34px shrink** the instant the user starts typing.
- **Root cause (empirically confirmed):** the Source placeholder ("Type or paste text in Chinese or
  English — Lucid detects the direction…", 71 chars) wraps to **2 lines** in the source column. With
  `field-sizing: content`, an empty textarea sizes to its placeholder → 2 lines ≈ 91px. Removing the
  placeholder via the DOM dropped the empty box to **57px** (proving the placeholder drives the tall empty
  state); restoring it returned it to 91px.
- **Why #26 exposes it:** under the old `min-h-[88px]`, the 88px floor nearly matched the 2-line
  placeholder (empty ≈91 → one-line 88, a ~3px change). Lowering the floor to `min-h-[56px]` unmasks the
  gap (empty 91 → one-line 57), turning a previously-imperceptible reflow into an obvious jump. So the
  regression is a direct, user-visible side effect of this feature.
- **Scope:** Source only. Polish Original/Draft placeholders are single-line, so their empty box already
  rests at 56px and there is no jump (56 → ~56).
- **Suggested fix directions (for triage, not applied here):** shorten the Source placeholder to one line,
  or pin the empty Source to the 56px floor (e.g. so the placeholder does not drive `field-sizing`), so
  empty == one-line as the design requires.

## Deferred
- **Built-CSS grep after `pnpm build` (plan DoD / Gate-2 Top-Risk 1).** Verified in dev, not the
  production bundle. The Tailwind-content-scan risk (that moving the `min-h-[56px]`/`min-h-[98px]` literals
  into a `.ts` constant would stop them being emitted) is nonetheless **resolved**: live
  `getComputedStyle().minHeight` returned `56px`/`98px`, which only happens if Tailwind emitted the rule
  AND it applied to the element — stronger than a static CSS grep. The literal `pnpm build` + `grep` step
  runs at the merge gate via `pnpm check:all`.
- **Phone tier (<600px, 50vh cap).** Not exercised — verification was scoped to desktop width (1400px).
  The `min-h-[56px]` constant is tier-agnostic (design: "same rule, no phone fork"), so phone resting
  height is also 56px; only the max-h cap differs (50vh), and that fragment is preserved verbatim and
  out of #26's scope.
