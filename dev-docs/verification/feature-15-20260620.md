# Feature #15 — Cross-device E2E-encrypted provider config + API key

Final-WI acceptance. A passphrase derives the key that AES-GCM-encrypts the provider config **and keys**
(built-in + custom) before they leave the device; the self-hosted server stores ciphertext only; a returning
device unlocks with the passphrase and decrypts locally. This is what makes the user's **DeepSeek custom-provider
key** persist across refresh and devices (the original motivation — keys were in-memory-only and vanished on
refresh). Verified by a **real-server browser E2E** (real crypto + real server + the real gate UI) plus the
exhaustive unit/integration suite (1402 tests, 100% gated coverage) and the two-lens + round-2 Gate-4 audits.

## Observations

- The blob on the server is the AES-GCM envelope only (`PBKDF2-SHA256`, 600000 iterations) — confirmed live that
  no `vendor`/`apiKeys`/`sk-`/model string appears in cleartext. The user's keys genuinely never leave the device
  unencrypted, and the server (which the user self-hosts) cannot read them.
- `localhost` is a browser secure context, so the real `crypto.subtle` ran (no mock) — the encrypt/decrypt path
  is exercised end-to-end, not just unit-mocked.
- A reload requires the passphrase again (it is never persisted) — the cross-device behavior, proven on one box.
- The codec was extended beyond the original (pre-#10) `SyncableConfig` so the user's **custom** DeepSeek key
  syncs, not only built-in keys — without that, #15 would not have solved the reported problem.
