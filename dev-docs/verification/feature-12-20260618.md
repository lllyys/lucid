# Feature #12 verification — persist provider config across reloads (keys stay in-memory)

Gate-5b final acceptance, run against the merged build on `main` (commit `1a1547f`, v0.8.0) served by
`vite dev` and driven via headless-Chromium CDP. Real `window.localStorage` (unlike the test runner,
where it is a no-op), so this exercises the actual `persist` middleware wiring + the `lucid.provider`
disk format end-to-end.

## Acceptance criteria

| Criterion (from the plan) | Result | Evidence |
| --- | --- | --- |
| Active vendor persists across reload | pass | Seeded a `{state:{vendor:'custom',…},version:1}` blob → reload → the toolbar provider switcher trigger rehydrated to **"Custom"** (`afterReload: "Custom"`). |
| Per-vendor model + custom base URL persist | pass | The seeded blob carried `models.custom` + `baseUrl`; the store rehydrated from it (vendor restored, `merge` re-derives the model mirror). Round-trip + mirror re-derivation also unit-proven (`providerStore.test.ts` round-trip test). |
| **API keys are NEVER written to disk (rule 65 §5)** | pass | After saving a valid Anthropic key via Settings, the actual persisted blob was `{"state":{"vendor":"anthropic","models":{…},"baseUrl":""},"version":1}` — **no key**, no `apiKey`/`apiKeys` field present. |
| No key rehydrates from disk | pass | The seeded rehydrate blob contained no key; `mergeProvider` sources `apiKey`/`apiKeys` only from fresh `current` (unit-proven: "NEVER rehydrates API keys" test). |
| Corrupt/unknown persisted data never crashes boot | pass | `createSafeJSONStorage` discards non-JSON/oversized; `mergeProvider` skip-bad-fields + membership-guards the vendor (unit-proven: non-object / non-record models / unknown vendor / non-string fields — no throw). |

## Observations

- The §5 guarantee is verifiable at the **storage byte level**: the persisted blob is exactly the
  `{vendor, models, baseUrl}` allowlist — a saved API key never appears in `localStorage`.
- A model key whose shape fails Anthropic validation is silently not saved (no persist write) — the
  first verification run surfaced this (fake key → empty localStorage), confirming the key-validation
  gate sits in front of persistence.
- The unit suite tests the pure helpers directly because the Node test runner has no real localStorage;
  this browser pass closes that gap end-to-end on the real middleware.
