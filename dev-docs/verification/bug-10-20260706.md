# Bug #10 — custom endpoint works via curl but fails in-app with a misleading "Provider unavailable"

Browser verification of the fix (v0.23.2): a browser fetch that never connects (CORS / mixed-content / offline —
all `TypeError: Failed to fetch`) now maps to a distinct **`unreachable`** error with actionable guidance,
instead of the misleading `providerDown` ("temporarily unavailable, try again").

## Environment
- Built the production app (`pnpm build` → `dist/`, v0.23.2, from the fix branch); served via a token-free
  single-origin `@lucid/server` (port 8807, `/tmp` DB). Headless Chrome via raw CDP; fresh profile.
- Seeded `localStorage['lucid.provider']` (v2) with an active custom provider whose base URL is
  `http://127.0.0.1:9999/v1` (nothing listening → the browser fetch is **refused** → `TypeError` — the same
  TypeError class a CORS block or an HTTPS→http mixed-content block throws). Typed "hi" → Translate.

## Observations
- The banner (screenshot `/tmp/lucid-228-banner.png`) renders the new title + actionable message on the
  already-designed error surface — no new visual state, just corrected copy via the existing `ErrorKind → title`
  Record. This directly resolves the reported scenario: the user's qwen3.6 endpoint (reachable via curl,
  CORS-blocked in the browser) would now show this actionable message pointing at CORS instead of the misleading
  transient-outage message.
- The repro uses connection-refused (127.0.0.1:9999) rather than a live CORS-less server because both produce
  the identical browser `TypeError: Failed to fetch` that the fix keys on; a real CORS/mixed-content block is
  the same code path (the browser collapses them into one opaque TypeError).
- **Diagnosis-only:** lucid still cannot *reach* a CORS-less endpoint from the browser (that needs a same-origin
  proxy — a separate future feature, rule 65 §5). This fix makes the failure honest and actionable; the user's
  path to actually using the endpoint is to enable CORS on the server (or a proxy).
