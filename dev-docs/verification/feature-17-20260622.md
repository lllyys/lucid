# Feature #17 — Workspace scroll model (resolved by feature #16)

#17 ("scroll the whole interface, not inside it") was the scroll-model **decision** that #16's responsive
design resolved. The committed `lucid-responsive-mobile-layout` design (which the user produced via
claude.ai/design) chose to **keep the app-shell + inner-scroll model** rather than switch to whole-page
(`document`) scroll, because a pinned footer/toolbar (so the Run button stays thumb-reachable) is
incompatible with `body` scroll. The user's underlying complaint — content scrolling *inside* nested
per-card regions rather than as one — is resolved by making **`<main>` the single scroll region** with the
chrome pinned, and by **dropping PolishPanel's nested input-column scroll on phone**.

## Acceptance criteria

| # | Criterion | How verified | Result |

| 1 | One scroll region, not nested per-card scrolls | `<main overflow-auto>` is the single region; PolishPanel input column is `min-[600px]:overflow-auto` (no phone nested scroll); unit asserts no nested Polish scrollbar on phone | pass |
| 2 | Chrome pinned (footer/toolbar/switcher don't scroll away) | design app-shell (`h-dvh` + `overflow-hidden` root); CDP: no horizontal overflow, pinned footer | pass |
| 3 | Reconciled with #13 auto-expanding editors | `field-sizing-content` + tier-scoped cap (`50vh` phone / `88vh` desktop) preserved | pass |
| 4 | No page-scroll regression on desktop | desktop unchanged (CDP desktop tier; existing tests green) | pass |

## Observations
- This is a "resolved by design decision" feature: the literal "page scroll" was NOT implemented because
  the user's own committed design chose app-shell inner-scroll; the single-content-column scroll satisfies
  the "scroll as one, not inside it" intent. Decided in `dev-docs/plans/20260622-feature-16-responsive-layout.md`
  (the #17 scroll-model section) and verified together with #16.
