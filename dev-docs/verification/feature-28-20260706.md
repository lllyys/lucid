# Feature #28 — same-origin LLM proxy (the deeper fix for bug #10)

End-to-end acceptance with **real processes** (not mocked fetch): the built app served token-free single-origin by
the real `@lucid/server`, a real mock OpenAI-compatible upstream (a private `http://` endpoint the browser cannot
reach directly), and the allow-list set via `PROXY_ALLOWED_UPSTREAMS`. Verified both the server relay (curl) and
the **browser-networking claim in real Chrome** — the part unit tests (which mock `fetch`) cannot cover.

## Setup
```
# mock upstream: POST /v1/chat/completions → SSE "Hola"/" mundo"/[DONE]
node mock-upstream.mjs 8901
# the app + proxy, token-free single-origin, allow-list = the mock
STATIC_DIR=$PWD/dist PROXY_ALLOWED_UPSTREAMS=http://127.0.0.1:8901/v1 PORT=8805 node server/dist/index.js
```
Startup log confirmed: `LLM PROXY ENABLED — /proxy relays to allow-listed upstreams: http://127.0.0.1:8901/v1`.

## Acceptance criteria

| # | Criterion | Verification | Result |
|---|---|---|---|
| 1 | The browser CANNOT reach a CORS-less/private upstream directly (bug #10) | Real Chrome, from the served page's JS: `fetch('http://127.0.0.1:8901/v1/chat/completions')` → **`TypeError: Failed to fetch`** (blocked) | ✅ pass |
| 2 | The same-origin `/proxy` relay streams the upstream SSE back in a real browser | Real Chrome: `fetch('/proxy', {POST, x-lucid-proxy-upstream, body})` → 200, `content-type: text/event-stream`, body = `data: …"Hola"… "mundo"… [DONE]`; the mock logged a browser-originated hit | ✅ pass |
| 3 | `GET /proxy` advertises the operator allow-list (capability) | Real Chrome + curl: `{ "upstreams": ["http://127.0.0.1:8901/v1"] }` | ✅ pass |
| 4 | An UNLISTED upstream is rejected (SSRF allow-list) with no target echo | curl `POST /proxy` unlisted → **403 `{"error":"forbidden upstream"}`**; the upstream was never fetched | ✅ pass |
| 5 | The client key is forwarded to the allowed upstream but NEVER logged (rule 65 §5) | mock log shows `auth=Bearer sk-…-KEY` (forwarded, Host rewritten); server log: **0** occurrences of either key | ✅ pass |
| 6 | Direct-by-default (no regression) | `shouldProxy` false for empty allow-list / unlisted / built-in / token-set / non-single-origin → the existing direct path (unit-tested, both call sites) | ✅ pass (audit + unit) |
| 7 | `pnpm check:all` green incl. 100% server coverage on the SSRF-critical surface | `proxy.ts` + the `/proxy` route 100%-gated, wired into `check:all` | ✅ pass |

## Commands / observations
- Server relay (curl): allowed → SSE streamed token-by-token; unlisted → 403 no-echo; the mock logged the
  forwarded Authorization + rewritten Host; the server log never printed either key.
- Browser (real Chrome at the served origin): the **direct** cross-origin fetch to the upstream failed with
  `TypeError: Failed to fetch` (the exact bug-#10 symptom), while the **same-origin `/proxy`** GET advertised the
  allow-list and POST streamed the upstream SSE back — proving the proxy mechanism removes the browser block.
- The client routing decision (token-free single-origin + custom + listed → proxy; else direct) is covered
  deterministically by `proxyRoute.test.ts`, the `openaiCompatibleProvider`/`createProvider` proxy-branch tests,
  and `usePanelRun`/`useTestConnection` tests (Gate-4 verified).
