("Draft to polish") header — the sibling of feature #23's Original Clear, now on the busier
Draft header. Design bundle `dev-docs/designs/lucid-draft-clear/` (issue #226 / GH #225).

## Method
```bash
pnpm dev --port 5233 --strictPort                 # Vite v7.3.5 on localhost:5233 (owned; PID killed at end)

```
- Behavior asserted, not model wording (rule 65 §8 / 66 §4). The Clear control is provider-independent;
  no LLM call is involved.
- Draft value set through the native `HTMLTextAreaElement.value` setter + a bubbling `input` event so
  React's `onChange` fires (a controlled component).
- Hover / focus-visible colors read via CDP `CSS.forcePseudoState` + `CSS.getComputedStyleForNode` (the
  DevTools force-element-state path), compared against the live-resolved token rgb values
  (`--text-tertiary` / `--text-color` / `--accent-ink`). Resting color cross-checked with JS
  `getComputedStyle`.

## Deferred / caveats
- **"Clear fires no LLM call under auto-run" (non-arming guarantee)** — delegated to the unit suite
  (`PolishPanel.test.tsx` / `DraftCard.test.tsx`), per the brief: provider config in a headless browser
  is automation-hostile, and auto-run requires a configured key. In this run auto-run was OFF, so no
  debounced polish could fire regardless. The negative is proven headlessly, not in-browser.
- **RTL context** — lucid ships only an English (LTR) locale today; the DraftCard header does NOT set
  `dir` from the draft's content language (only the textarea flips via `dir="auto"`). The header mirror
  was therefore exercised by forcing `document.documentElement dir="rtl"` — the exact cascade an RTL UI
  locale would produce. The flex/logical layout mirrored correctly and Clear still led; this is a
  faithful simulation of the design's RTL state, noted here because it is a forced (not locale-driven)
  context.
- **Production bundle** — verified in dev, not `pnpm build`/`preview`; the literal build + `pnpm check:all`
  run at the merge gate.
