# Bug #11 — polish result flashes then disappears (keywords-change effect reset on a same-content reference change)

Verification-exception (fix-issue Phase 9b): the defect + fix are **pure component logic** — the keyword-change
effect in `PolishPanel`. The user-visible symptom (the `done` polish result being wiped) is produced entirely by
that effect calling `resetPolish()`; `PolishResult` renders off `op.status`. The fix is verified by a
**deterministic RED→GREEN component test** that drives the real `PolishPanel` effect + the real
`operationStore`/`polishKeywordsStore` (not stubs). A real browser polish needs a live provider **key** (held
in-memory only, rule 65 §5 — impractical to seed in CDP), so the effect-boundary test is the high-fidelity
verification.

## Observations
- Root cause: `PolishPanel.tsx` keyed the "keyword changed → invalidate the polish result" effect on the
  keywords array **reference**; `src/lib/sync/reconcile.ts` re-applies keywords into the store every cycle
  (fresh array, identical content), so polish `done` → `useAutoRecordTask` records the task → the dirty task
  triggers a sync cycle → reconcile re-sets keywords → the effect falsely fired `resetPolish()`. Latent (effect
  + sync both pre-existed); surfaced by sync going default-on (#21) — polish lost its result on ~every run.
- Fix: compare a JSON value-key. Same content → same key → the effect doesn't re-run → the result persists; a
  real change → different key → reset + re-arm (behavior preserved).
