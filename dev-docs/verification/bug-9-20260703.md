# Bug #9 — starred list showed the same word twice (dedup keyed on context)

Verification-exception path (fix-issue Phase 9b): the defect is a **pure store-logic** dedup bug. The
user-observable symptom (a duplicate row in ★ Starred) is produced entirely by the store's `star()` action +
`sameContent` dedup; `StarredView` renders `store.items` verbatim (already browser-verified in #22/#24/#25). The
fix is therefore verified by a **deterministic high-fidelity test at the real subsystem boundary** — it drives
the actual `star()` action + the actual `sameContent`, not a stub.

## Observations
- The RED→GREEN transition on `star()` (the real store action, no stub) is the highest-fidelity verification for
  this defect — the dedup decision is made there, and `StarredView` is a pure render of `store.items`.
- Root cause: `context` was in the dedup tuple for same-lookup idempotency (#22 Gate-2 M3), but it's populated
  only for word lookups, so the same word from a different sentence had a different `context` and evaded dedup.
- Accepted tradeoffs (Gate-4 Lows): polysemy collapse (two senses of one word → one entry, first wins — intended
  per "one entry per word + direction"); legacy localStorage dupes persist until unstarred (no silent delete).
